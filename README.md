# Developer Questions Mining Pipeline

A research pipeline that mines GitHub repositories to build a labelled dataset of **developer information needs** — real questions developers ask about their own codebase, with ground-truth answers extracted from issue threads, discussions, PR comments, commits, and CI history.

The dataset is built around a 78-question taxonomy (Q1–Q78) covering 11 categories: People & Awareness, Code Changes, Work Item Progress, Broken Builds, Test Cases, Pull Requests, Bug Management, CI/CD, Security & Quality, Onboarding, and Cross-Artifact Synthesis.

---

## How it works

The pipeline has two independent paths that produce complementary outputs:

```
GitHub API
    │
    ├── miners/                    ← structured artifact extraction
    │   ├── issues.py              → issues.jsonl, duplicate/assignment ground truth
    │   ├── pull_requests.py       → pull_requests.jsonl, review timings
    │   ├── commits.py             → commits.jsonl, szz_pairs.jsonl (regression ground truth)
    │   ├── ci_runs.py             → ci_runs.jsonl, ci_workflow_failures.jsonl
    │   ├── contributors.py        → contributors.jsonl, file_experts.jsonl, file_ownership.jsonl
    │   └── mine_threads.py        → raw_threads.jsonl (issues + Discussions as flat text)
    │
    ├── classification/
    │   └── classify.py            ← two-stage LLM classifier (local Ollama)
    │       Stage 1: broad category (A–K) → Stage 2: specific question ID (Q1–Q78)
    │       Output: natural_qa_pairs.jsonl
    │
    └── miners/qa_builder.py       ← structured ground-truth builder
        Extracts deterministic (question, ground_truth, artifacts_needed) triples
        from the structured JSONL files.
        Output: qa_pairs.jsonl
```

### natural_qa_pairs.jsonl
Produced by the LLM classifier. Each record is a real Q&A exchange extracted verbatim from a GitHub thread, tagged with a question ID, confidence score, reasoning, and the full original thread for human review.

### qa_pairs.jsonl
Produced by the structured builder. Each record is a deterministic question/answer triple with exact ground truth (e.g. the SHA of the inducing commit, the assignee login) and the artifacts an LLM agent would need to answer it. Used for automatic evaluation.

```
pipeline/
├── config.py                  # Repos list, GitHub tokens, output path, thresholds

├── miners/
│   ├── issues.py

│   └── qa_builder.py

│   └── classify.py

    └── taxonomy.py            # Full 78-question taxonomy string for prompts

output/
└── <owner>__<repo>/
    ├── issues.jsonl
    ├── pull_requests.jsonl
    ├── commits.jsonl
    ├── szz_pairs.jsonl
    ├── ci_runs.jsonl

    ├── raw_threads.jsonl
    ├── natural_qa_pairs.jsonl  ← LLM-classified Q&A pairs (human-reviewable)

├── app.py                     # FastAPI review server
└── templates/index.html       # Single-page review UI

verified_state.json            # Human verification decisions (auto-created)
verified_qa_pairs.jsonl        # Exported accepted pairs (generated by review UI)
```

## Setup

**Conda environment** (stored at `/local/home/amamun/envs/devqa`):

```bash
# Activate
source activate /local/home/amamun/envs/devqa
# or use the full path
/local/home/amamun/envs/devqa/bin/python ...
```

**Environment variables** — create a `.env` file in the project root:

```
GITHUB_TOKEN=ghp_your_token_here
# Multiple tokens for higher rate limits:
# GITHUB_TOKENS=ghp_token1,ghp_token2
```

**Ollama** must be running locally for the LLM classification step:

```bash
ollama serve
ollama pull qwen3.6:latest   # or whichever model is set in ollama_client.py
```

## Running the pipeline

```bash
cd pipeline

# Mine everything for all repos in config.py
python run_all.py

# Mine a single repo
python run_all.py --repo psf/requests

# Skip CI mining (slow for large repos)
python run_all.py --repo psf/requests --skip-ci

# Only rebuild QA pairs (mining already done)
python run_all.py --only-qa

# Mine threads only (needed before classification)
python miners/mine_threads.py --repo psf/requests

# Classify threads with the LLM (requires Ollama)
python classification/classify.py --repo psf/requests

# Re-classify with a different model
python classification/classify.py --repo psf/requests --stage1-model mistral:7b

# Re-classify everything from scratch
python classification/classify.py --repo psf/requests --force

# Only classify the first 100 threads (for testing)
python classification/classify.py --repo psf/requests --limit 100
```

## Reviewing and validating output

### CLI summary
```bash
python summarize_qa_pairs.py
```
Prints a table of all question IDs found across all repos, with instance counts.

### Review UI
```bash
cd review_ui
/local/home/amamun/envs/devqa/bin/python app.py
# → http://localhost:8765
```

The UI lets you:
- Browse all `natural_qa_pairs.jsonl` records with filters (repo, question ID, status, text search)
- See full question/answer, confidence score, reasoning, and original thread
- Accept or reject each pair with an optional note
- Export accepted pairs to `verified_qa_pairs.jsonl`

Keyboard shortcuts: `j`/`k` navigate, `a` accept, `r` reject, `u` reset to pending.

Verification state persists in `verified_state.json` at the project root.

---

## Adding a new repository

1. Add `"owner/repo"` to `REPOS` in `pipeline/config.py`
2. Run the pipeline: `python run_all.py --repo owner/repo`
3. Run classification: `python classification/classify.py --repo owner/repo`
4. Review results in the UI at `http://localhost:8765`

## Output formats

All files are newline-delimited JSON (`.jsonl`) — one record per line.

**natural_qa_pairs.jsonl** — key fields:

| Field | Description |
|---|---|
| `source` | `"issue"` or `"discussion"` |
| `repo` | `"owner/repo"` |
| `number` | Issue/discussion number |
| `question_id` | Taxonomy ID, e.g. `"Q31"` |
| `question_text` | Verbatim question from the thread |
| `answer_text` | Verbatim answer from the thread |
| `answer_author` | GitHub login of the answerer |
| `answer_is_accepted` | Whether the issue closed after this answer |
| `confidence` | Stage1 × Stage2 confidence product (0–1) |
| `reasoning` | One-sentence LLM reasoning |
| `stage1_category` | Broad category letter (A–K) |
| `thread_text` | Full original thread (for review) |
| `stage1_model` / `stage2_model` | Models used |

**qa_pairs.jsonl** — key fields:

| Field | Description |
|---|---|
| `question_id` | Taxonomy ID |
| `question` | Templated question string |
| `ground_truth` | Exact answer (login, SHA, count, etc.) |
| `artifacts_needed` | Data an agent needs to answer |
| `tier` | Difficulty tier (1 = straightforward) |
| `repo` | Source repository |

## Rate limits

The GitHub REST API allows 5,000 requests/hour per token. `utils/github_client.py` checks remaining quota before each request and sleeps automatically. Use multiple tokens via `GITHUB_TOKENS` in `.env` for higher throughput.
